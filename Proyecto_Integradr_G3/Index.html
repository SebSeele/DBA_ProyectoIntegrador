1. Objetivos del Proyecto

Objetivo General

Diseñar y desplegar una arquitectura distribuida integrando Oracle, MariaDB y SQL Server para simular el ecosistema de "SeeleStore", garantizando interoperabilidad mediante Docker y Python.

Objetivos Específicos

  • Fase 1: Despliegue manual y hardening en VMs Linux.
  • Fase 2: Evolución a Microservicios con Docker Compose.
  • Interconexión: Middleware ETL en Python con automatización.
  • Validación: Interfaz CLI para auditoría transaccional.

2. Caso de Negocio: "SeeleStore"

Una plataforma de e-commerce requiere gestionar altos volúmenes de datos distribuidos en tres áreas críticas:

MariaDB

Catálogo & Clientes
Lectura rápida de perfiles y productos.

Oracle 21c

Transaccional (OLTP)
Núcleo financiero seguro para órdenes de compra.

SQL Server

Analítica (OLAP)
Reportes financieros consolidados (Data Mart).

3. Fase 1: Infraestructura Tradicional (VMs)

Despliegue manual sobre Ubuntu Server 22.04 LTS. Se priorizó el control granular y la seguridad.

  • Configuración de Red: IPs estáticas para comunicación entre nodos.
  • Oracle Hardening: Configuración manual de listener.ora en puerto 1522 y Tablespaces dedicados.li>
  • MariaDB Security: Script mysql_secure_installation y cambio a puerto 3307.
  • SQL Server: Configuración vía mssql-conf en puerto 1435
"Esta fase estableció la línea base de seguridad y validó la conectividad TCP/IP."

4. Fase 2: Evolución a Microservicios

Migración a Docker sobre WSL2 para optimización de recursos y portabilidad (IaC).p>

Docker Compose

  • Red Aislada: red_fica
  • Oracle Container: Modo privileged y gestión de ulimits.
  • Middleware: Contenedor Python dedicado (Worker).
services:
  oracle:
    image: gvenzl/oracle-xe:21-slim
    ports: "1522:1521"
  middleware:
    build: ./interconnector
    depends_on: [oracle, mariadb]

5. Capa de Interconexión (Middleware)

Desarrollada en Python 3.9, actúa como el cerebro operativo del sistema.

Automatización (ETL Daemon)

  • Wait-for-DB: Lógica de espera inteligente al arranque.
  • Data Seeding: Generación sintética con librería Faker cada 60s.
  • ETL: Extracción de Oracle -> Transformación (Pandas) -> Carga en SQL Server.

Gestión Manual (CLI)

  • Interfaz de comandos interactiva.
  • Permite inyección de datos controlada.
  • Auditoría en tiempo real.

7. Conclusiones

  • Heterogeneidad Exitosa: Integración estable de tecnologías dispares (PL/SQL, T-SQL) usando Python como vector.
  • Evolución Tecnológica: El paso de VMs a Docker redujo el consumo de recursos y eliminó conflictos de dependencias ("Works on my machine").
  • Integridad Transaccional: La configuración de Tablespaces y tipos de datos decimales protegió la precisión financiera del sistema.
  • Trabajo en Equipo: La división de roles (Infraestructura, Backend, Datos) permitió un desarrollo paralelo eficiente.

8. Recomendaciones (Roadmap)

Escalabilidad

Migrar de Docker Compose a Kubernetes para auto-scaling y self-healing.

Desacoplamiento

Implementar colas de mensajería (Kafka/RabbitMQ) para una arquitectura orientada a eventos[cite: 202].

Seguridad

Usar HashiCorp Vault para la gestión de secretos y rotación de credenciales.

Observabilidad

Integrar Prometheus + Grafana para monitoreo de métricas de negocio en tiempo real.
